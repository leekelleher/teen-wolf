uComponents

I went back to my uComponents roots... I took a look at some of the simpler Property Editors that we had, that no one else appeared to have modernised. I recalled one called RenderMacro, which was a readonly editor, that rendered a macro in the backoffice to appear as an instructional/editorial note - or a breadcrumb trail.
It was cathartic.
uComponents RenderMacro was one of those were people wasn't expecting it, nor demanding it. I felt safe to explore.

STOP!

DEMO TIME!

Render Macro

then prototyped how that would work in v8. Then making Render Macro, meant that I needed a Macro Picker so I looked around the Umbraco core source to see what I could find... no property editor as such, there was a Macro Picker for TinyMCE, and one for a Grid Editor, and another for the Template editor... but not property-editor, the overlay was there, just the UI editor was missing.

Macro Picker

so I made that... now the Macro could be selected and configured. But then the Macro Picker would need a way to restrict the available Macros... so use the Macro Picker inside itself

Entity Picker

but wait, no, this wasn't a way to configure a Macro (with parameters), I just wanted to reference the Macro, (say by UDI), so it was really an Entity Picker that was needed... so I made that. Where the Entity Picker could be configured to select from different Umbraco entity types. I found the `entityResource` API, where you could pass through the `entityType`. I found the Umbraco "entitypicker" which was a dropdown, with multi-select option ... it wasn't very slick, wasn't very "v8".

Data List

At this point, I had wrangled the Item Picker overlay (from Umbraco), that I could populate it with whatever I wanted... (as long as the item had a name/value/icon/description), then we could have a picker for anything. So I made it extensible... having various data sources - Enum, SQL, XML, JSON, Umbraco entities - and it was pluggable too! ... But wait, if I made the data source extensible - why not the editor UI? Why have a separate Property Editor if I wanted to populate, say, a checkbox list or dropdown list or radiobutton list from one of those data sources? So I made that extensible too - and Data List was born. Oh no, I'd killed nuPickers!
eek, love you Hendy! but looking back over my uComponents ideas, there was an idea for Data List that pre-dated nuPickers with the involvement of Hendy, Tom Fulton, et al.

Cascading Dropdown List

In making Data List, the component that powered the custom data source / list editor provider became its own thing� Configuration Editor. Give it list of C# classes (of a common interface) and it would open an overlay editor with those `ConfigurationField` options. In prototyping some of the data sources, other Property Editors were created. E.g. with the Enum data source, I needed an interface that would cascade a selected value, e.g. an assembly name, to the next interface, to select the enum type. I made this as a series of dropdown lists� Cascading Dropdown List editor became a thing - provide a list of API endpoints, which will call the next endpoint with the selected value(s).

Icon Picker

Another was Icon Picker, which was needed for the general Item Picker, so if a custom data source didn't provide an icon, a default one could be used.

Notes

For some of the data source and list editors, I wanted to display friendly helpful notes, so I created a basic Notes property editor - a lightweight version of the uComponents Notes and Marc Goodson�s uEditorNotes package.

My relationship with...

ENTITLEMENT

The "take, take, take" attitude was eating away at me. How can I set the expectations?
I had to understand what Entitlement meant.
I found this... "Entitlement is the separation of rights from responsibility."
Open source communities and ecosystems can't remain stable and sustainable without respecting that rights and responsibility are inseparable.

My relationship with...

RIGHTS AND RESPONSIBILITIES

As the developer of the software, I am granting you rights to use the software.
You are being empowered.
As a user, the responsibility of using that software is yours not mine. You have been empowered!.
The mistake that is often made is that users tend to see themselves as customers of Open Source software.
You are not a customer unless you are paying for a service. Business critical issues are your responsibility not the Open Source developer.
